# Concept to WIL Skill

작업 중 발생하는 **개념 질문**과 **설계 고민**을 설명하고, 옵시디언 WIL 문서 기록은 `wil-writer` 에이전트에 위임한다.

```
타입 A (개념 질문): 설명 → wil-writer 에이전트에 저장 위임
타입 B (설계 고민): 논의 → 결론 도출 → wil-writer 에이전트에 저장 위임
```

---

## 타입 A: 개념 질문

사용자가 모르는 개념에 대해 질문하면 설명 후 기록을 위임한다.

### 설명 원칙

- 핵심을 먼저, 세부 사항은 나중에
- 비유나 대비를 활용하여 이해하기 쉽게
- 코드 예시가 도움이 되면 반드시 포함
- 사용자의 현재 작업 맥락과 연결하여 설명

---

## 타입 B: 설계 고민

작업 중 설계 방향에 대한 의문이 제기되고, 논의를 통해 결론이 나면 기록을 위임한다.

### 대상

- 비즈니스 로직 순서에 대한 고민 (예: "재고 차감 전에 주문 생성해도 되나?")
- 연관관계/의존성 설계 결정 (예: "ID 참조 vs 엔티티 참조")
- 아키텍처/패턴 선택 (예: "Service에서 처리 vs Facade에서 처리")
- 유즈케이스/요구사항 해석 차이

### 저장 시점

- 고민이 제기된 시점이 아니라, **결론이 나고 코드/문서에 반영된 후**에 저장을 위임한다.
- 결론이 나지 않은 열린 고민은 저장하지 않는다.

---

## WIL 저장 위임

설명 또는 결론 도출이 완료되면, `wil-writer` 에이전트에 저장을 위임한다.

### 위임 방법

Task 도구로 `wil-writer` 에이전트를 호출하며, 프롬프트에 다음을 포함한다:
- **타입**: A(개념 질문) 또는 B(설계 고민)
- **제목**: 사용자의 질문 또는 고민 주제
- **본문**: 사용자에게 설명한 내용 전체 (마크다운 형식 그대로)

### 위임 시점

- 사용자에게 설명을 **먼저 보여준 후**, 백그라운드로 에이전트를 실행한다.
- 에이전트 완료를 기다리지 않고 대화를 계속한다.

---

## 스킬 적용 판단 기준

### 적용 O

**개념 질문 (타입 A)**
- ":: 문법이 뭐지?"
- "toList()랑 collect 차이가 뭐야?"
- "dirty checking이 무슨 뜻이야?"
- "guard() 메서드가 뭐하는 거야?"

**설계 고민 (타입 B)** — 결론이 난 후 기록
- "재고 차감 안 했는데 주문이 생성되는 게 맞나?" → 순서 변경 후 기록
- "User-Order 연관관계를 맺어야 하나?" → 결정 후 기록
- "이 로직은 Service에 둘까 Facade에 둘까?" → 결정 후 기록

### 적용 X

- 프로젝트 구조나 코드 위치에 대한 질문 ("이 파일 어디 있어?")
- 작업 지시 ("리뷰해줘", "수정해줘")
- 요구사항 분석 요청
- 결론이 나지 않은 진행 중인 논의
