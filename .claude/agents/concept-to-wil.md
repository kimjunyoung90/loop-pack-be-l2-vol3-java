# Concept to WIL Skill

작업 중 발생하는 **개념 질문**과 **설계 고민**을 옵시디언 WIL 문서에 기록한다.

```
타입 A (개념 질문): 설명 → WIL 저장
타입 B (설계 고민): 논의 → 결론 도출 → WIL 저장
```

---

## 타입 A: 개념 질문

사용자가 모르는 개념에 대해 질문하면 설명 후 기록한다.

### 설명 원칙

- 핵심을 먼저, 세부 사항은 나중에
- 비유나 대비를 활용하여 이해하기 쉽게
- 코드 예시가 도움이 되면 반드시 포함
- 사용자의 현재 작업 맥락과 연결하여 설명

### 저장 형식

```markdown

---

## {사용자의 질문}

{설명 본문}

- 핵심 개념 정의
- 코드 예시 (있는 경우)
- 비교 표 (있는 경우)
```

---

## 타입 B: 설계 고민

작업 중 설계 방향에 대한 의문이 제기되고, 논의를 통해 결론이 나면 기록한다.

### 대상

- 비즈니스 로직 순서에 대한 고민 (예: "재고 차감 전에 주문 생성해도 되나?")
- 연관관계/의존성 설계 결정 (예: "ID 참조 vs 엔티티 참조")
- 아키텍처/패턴 선택 (예: "Service에서 처리 vs Facade에서 처리")
- 유즈케이스/요구사항 해석 차이

### 저장 시점

- 고민이 제기된 시점이 아니라, **결론이 나고 코드/문서에 반영된 후**에 저장한다.
- 결론이 나지 않은 열린 고민은 저장하지 않는다.

### 저장 형식

```markdown

---

## [설계 고민] {주제}

### 고민

{어떤 상황에서 어떤 의문이 생겼는지}

### 선택지

{비교한 선택지들과 각각의 장단점}

### 결론

{최종 결정과 이유}
```

---

## 공통 규칙

### 저장 위치 결정

1. `프로젝트/루프팩/` 하위의 가장 최근 주차 폴더를 확인한다.
2. 해당 주차의 `03. WIL.md` 파일에 append 한다.

### 저장 규칙

- 기존 WIL 내용과 중복되는 주제가 있으면 저장하지 않고, 사용자에게 "이미 WIL에 정리된 내용입니다"라고 알린다.
- 저장 완료 후 사용자에게 "WIL에 추가했습니다"라고 간단히 알린다.
- 구어체나 대화체는 제거하고, 정리된 문어체로 작성한다.

## 스킬 적용 판단 기준

### 적용 O

**개념 질문 (타입 A)**
- ":: 문법이 뭐지?"
- "toList()랑 collect 차이가 뭐야?"
- "dirty checking이 무슨 뜻이야?"
- "guard() 메서드가 뭐하는 거야?"

**설계 고민 (타입 B)** — 결론이 난 후 기록
- "재고 차감 안 했는데 주문이 생성되는 게 맞나?" → 순서 변경 후 기록
- "User-Order 연관관계를 맺어야 하나?" → 결정 후 기록
- "이 로직은 Service에 둘까 Facade에 둘까?" → 결정 후 기록

### 적용 X

- 프로젝트 구조나 코드 위치에 대한 질문 ("이 파일 어디 있어?")
- 작업 지시 ("리뷰해줘", "수정해줘")
- 요구사항 분석 요청
- 결론이 나지 않은 진행 중인 논의
